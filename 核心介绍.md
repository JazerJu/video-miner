# 流式转录
## Streaming Whisper
该项目目前定位是 “低延迟分段转写” 而不是
  “完全实时流式转写”。它的“实时”体现在你一说完，结果几乎立刻出来，而不是在你说的过程中文字逐个蹦出来。

它的处理流程如下：
1. 监听与切分 (VAD):
    * 客户端（或本地脚本）使用 webrtcvad 实时监听麦克风音频。
    * 它以 30ms 为一帧进行分析。
    * 当检测到一段持续的语音（超过 50% 的帧是人声）时，开始录音。
    * 当检测到一段持续的静音（说话停顿）时，停止录音并把这段音频打包成一个完整的 WAV 数据块（Chunk）。
2. 发送与队列:
    * 这个完整的音频块被发送到 Redis 列表（分布式模式）或内部队列（本地模式）。
3. 转写:
    * 服务端从队列中取出这个完整的音频块。
    * 调用 faster-whisper 对这个独立的音频块进行一次性转写。

## FireredASR.
 是的，原理上存在对近期“不稳定”片段的再次转录和修正。

  虽然该项目并没有运行两个并行的解码器进行“竞争”，但它通过“回退-
  重预测”（Back-off and
  Repredict）的策略，实现了你所描述的效果：模型确实会丢弃上一次
  结果的末尾部分，并在下一次有了更多音频上下文时，对这部分时间段
  （即第 N-1 ~ N 秒）进行重新转录。

  以下是结合代码的具体原理分析：

  1. 核心机制：尾部截断 (The "Look-back" Strategy)

  在 fireredasr/models/fireredasr_streaming.py
  的代码逻辑中，有一个关键的处理：

   1 # 获取本次完整的识别结果 ys
   2 hyp = hyps[0][0]
   3 ys = hyp["yseq"]
   4
   5 # 关键：更新 ys_state 时，故意切掉了最后一个 token (ys[:-1]
   6 if len(ys) > self.least_ys_state_len:
   7     self.ys_state = ys[:-1] if len(ys) > 0 else ys

   * 当前时刻 (T)：假设模型识别出文本为 ["我", "爱", "北"]。
   * 状态保存：系统只把 ["我", "爱"] 保存为
     ys_state，丢弃了最后一个字 "北"。
   * 下一时刻 (T+1)：
       * 输入：ys_state (["我", "爱"]) + 新的更长音频。
       * 预测：模型被强制先接受 ["我", "爱"]
         是正确的，然后基于新的音频特征，重新预测接下来的字。
       * 修正：此时，模型可能会再次输出
         "北"（确认结果），也可能因为听到了后面的音（如“京”），
         发现之前的音其实不是“北”，从而修正为其他字。

  2. 为什么这样做？(Why?)

  这种策略是为了解决流式语音识别中的“不稳定尾部”问题：

   * 音频上下文缺失：在第 N
     秒时，刚说出的字处于音频的末尾，缺乏“右侧上下文”（未来的信
     息）。Transformer 模型非常依赖上下文来区分同音字（例如“jiao
     yi”是“交易”还是“交椅”）。
   * 抗噪与平滑：尾部的音频特征往往因为截断而不完整。
   * 通过“再转录”实现修正：等到第 N+1 秒时，第 N
     秒的音频就不再是尾部了，它有了后续的音频作为参考。此时让模
     型重新转录第 N 秒的内容，准确率会大幅提升。

  总结

  虽然不是“并行竞争”，但这是一个串行的“预测 -> 撤回 ->
  再预测”过程。

   * 第 N 秒：转录出 A B C。
   * 第 N+1 秒：系统撤回 C，保留 A B，要求模型基于 A B
     和新音频重新预测。模型可能输出 A B C D（确认 C
     正确），也可能输出 A B E F（修正 C 为 E）。

  这正是为什么你在运行 test.py 时，会看到结果类型中有 type:
  'changed'，且文本会随着时间发生变化的原因。


# TTS功能

### 1. 智能音频合成
- **基于字幕**: 自动读取视频的 SRT 字幕文件
- **时间轴对齐**: 配音严格按照字幕时间轴生成
- **自然语音**: 使用阿里云 CosyVoice-v2 先进 TTS 引擎

### 五层容错机制
1. **智能重试**: 失败自动重试 5 次，指数退避（2s → 4s → 8s...）
2. **进度保存**: 每 10 段自动保存，崩溃后自动恢复
3. **动态限流**: 根据 API 状态自适应调整请求间隔
4. **优雅降级**: 个别段落失败不影响整体输出
5. **详细日志**: 完整的处理日志和错误报告

### 高成功率保障
- **改进前**: ~5% 成功率（遇到网络错误即失败）
- **改进后**: ~95%+ 成功率（自动处理各种异常）
